# AI 페어 워크플로우 — 이 레포가 실제로 만들어진 방식 (W-3 사례 · W-4 글감)

> 이 문서는 "AI로 개발했다"는 자랑이 아니라, **AI에게 무엇을 맡길 수 있고 무엇을 맡길 수 없는지**를 이 레포에서 실제로 겪은 사례로 정리한 기록이다.
> 설계 판단·측정은 `design-notes.md`, 진행 상태는 `START-HERE.md` §3. 여기는 **협업 구조** 축이다.

## 1. 역할 분담 (선언이 아니라 제약에서 나왔다)

| 주체 | 맡는 것 | 왜 |
|---|---|---|
| **Claude Code** | 설계·측정·검증·기록, 코드 리뷰 | 아래 §3-4 참조 — **검증은 구조적으로 위임 불가**했다 |
| **Codex** (`/codex:rescue`) | 구현 (파일 신설·수정·테스트 작성) | 스펙이 명확하면 로직 정확도가 높다 (§3) |
| **독립 에이전트** | 자기 산출물 감사 (골든셋 검수) | 자기 채점·자기 해설의 편향 차단 (§4-6) |

**핵심**: 이 분업은 취향이 아니라 **환경 제약이 강제**했다. Codex 샌드박스는 이 레포에서 **gradle을 실행하지 못한다**(소켓·락 차단). 즉 컴파일·테스트·측정은 처음부터 Codex가 할 수 없는 일이었다 → "구현은 위임, **검증은 보유**"가 자연 귀결.

## 3. 배운 것

### ① 로직은 맞고, **경계에서 샌다**

11건 중 로직이 틀린 적은 없다. 샌 곳은 전부 **경계**였다 — 레포 컨벤션(금칙어), 프롬프트 문구, 브라우저 보안 컨텍스트, 파일 인코딩, 프레임워크 고유 제약(ES dims), 기동 시점 빈 그래프.

> **위임 리뷰의 값어치는 "알고리즘 검산"이 아니라 스펙에 안 적힌 환경·컨벤션 지식의 주입에 있다.**

### ② **타입이 강제하는 누락만** 반복을 면한다

같은 위임(C3-2)에서 누락이 두 번 났는데 성질이 정반대였다:

- `RagResult`에 케이스를 추가하자 **sealed + exhaustive switch**가 스코프 밖 2곳의 누락을 **컴파일 에러로 즉시** 드러냈다.
- 반면 **가드·지표 누락**은 아무도 안 잡았다. 컴파일도 테스트도 통과했다.

### ③ **이월한 부채는 이자가 아니라 복제로 자란다**

②의 가드·지표 누락을 "다음 Phase에서"로 이월했더니, **바로 다음 스텝(C3-3)이 새 LLM 호출부를 추가하며 똑같은 구멍을 복제**했다. 스펙에 안 적었으니 또 빠진 것이다.

> 횡단 관심사(장애 격리·관측)는 **타입이 지켜주지 않으므로 위임 스펙에 명시**하거나, **가드를 지나지 않으면 호출할 수 없는 구조**로 강제해야 한다.

### ④ 검증은 위임할 수 없다 — 그래서 분업이 성립했다

Codex는 자기 코드를 **컴파일조차 못 한다**(샌드박스 제약). 그래서 "Codex가 못 보는 유형"의 결함(#4·#6·#10)이 구조적으로 남는다. 이건 Codex의 실력 문제가 아니라 **관측 범위 문제**다.

> 위임의 안전성은 위임받는 쪽의 능력이 아니라 **위임하는 쪽이 무엇을 검증할 수 있느냐**로 정해진다.

### ⑤ 도구의 **실증된 약점**과 과제 요구가 충돌하면 위임처를 바꾼다

Codex의 PowerShell 툴링은 이 환경에서 한국어 UTF-8을 신뢰성 있게 다루지 못한다는 게 **연속 2건으로 실증**됐다(#10: BOM 삽입, 정상 JSON 오판). 그 직후 과제가 **"한국어 골든셋 의미 감사"**였다.

→ 관례(위임=Codex)를 깨고 **일반 에이전트**에 맡겼다. 기본값을 지키는 것보다 **실증된 실패 모드를 피하는 것**이 우선이다.

### ⑥ 자기 데이터·자기 채점·자기 해설은 **프레이밍이 샌다** ⭐

골든셋 4종은 전부 Claude 초안이고, 그걸로 Claude가 측정하고 Claude가 해설했다. 독립 감사 결과:

- **사실 오류 0건** — 데이터는 멀쩡했다.
- 대신 **해설이 세 군데 미끄러져 있었다**: (a) 한 항목을 골든셋에선 오답으로 채점하면서 본문에선 옹호 = **이중잣대** (b) 대표 사례로 인용한 문장이 **사실이 아님**(고유 키워드가 실제로 존재) (c) 1건 폭 마진에 "이론이 예측한 지점"이라는 **과잉 일반화**.

> 숫자는 재현되니 틀리기 어렵다. 미끄러지는 건 **"이 숫자가 무엇을 뜻하는가"** 쪽이다.
> **정직성은 측정 설계가 아니라 리뷰 구조로 지켜진다.**
> B2에서 *"1회 확인은 거짓 안심"*을 배웠다면, 여기선 ***"자기 해설은 거짓 확신"***을 배웠다.

## 4. 프로토콜 (사례에서 역산한 규칙)

**위임 전**

1. **실체 확인**(C2-1 교훈): 쓰려는 스타터·클래스·API가 **그 버전에 실제로 있는지** 먼저 확인한다. 없는 걸 전제한 스펙은 통째로 낭비된다.
2. 스펙에 반드시 포함: 읽을 파일 목록(추측 금지) · 시그니처 · **횡단 관심사(가드·지표·로깅)** · 테스트 전략 · **하지 말 것**(커밋·docs 수정·범위 밖 파일).
3. **카나리아를 심는다**: "이 파일이 바뀌면 범위를 넘은 것"이라고 명시 → 스코프 크립을 diff로 즉시 감지.

**위임 중**

4. 장시간 작업은 `--background`(#3의 교훈 — 포그라운드 타임아웃이 Codex를 죽인다).

**위임 후 (검증은 항상 Claude)**

5. 컴파일 → **전체 스위트** → diff 리뷰 순. 풀 컨텍스트 부팅이 단위테스트가 못 잡는 걸 잡는다(#6).
6. **재위임은 최대 2회.** 사소한 수정(컴파일 에러·컨벤션·인코딩)은 왕복 비용이 더 크므로 직접 고친다.
7. **Codex의 보고를 믿지 말고 확인한다** — #10에서 Codex는 멀쩡한 파일을 "invalid"라고 보고했다. 그 말을 믿고 "고쳤다면" 정상 데이터를 망가뜨렸을 것이다.

**측정·데이터**

8. 측정치가 이미 공개된 데이터를 감사할 땐 **읽기 전용**. 데이터를 고치면 기록된 숫자가 조용히 무효가 된다 → 보고 → 사람이 판정 → 재측정.
9. 자기가 만든 평가 데이터는 **독립 감사**를 붙인다(§3-6).

## 5. 도구 아티팩트

이 워크플로우를 레포에 **보이게** 만든 것들:

- **`CLAUDE.md`** — 프로젝트 규칙(빌드·컨벤션·워크플로우)을 AI가 매 세션 읽게.
- **커밋 가드 2겹** (W-1) — git `pre-commit` 훅 + Claude Code `PreToolUse` 훅. 스테이지된 **추가 줄만** denylist와 대조(기존 표현 재검사 안 함). 실제로 #7에서 **작동**했다.
- **커스텀 커맨드** (W-2) — `.claude/commands/brief.md`(세션 시작 브리핑), `.claude/commands/record.md`(진행 기록·정직성 규칙·커밋). 둘 다 **실제 반복 작업에서 역산**했다: 모든 세션이 "두 문서 읽고 이어서"로 시작했고, 진행 기록은 한 세션에 10번 수작업했다.
- **MCP 서버** (C4) — 검증된 검색 자산을 표준 프로토콜로 개방(`search_wiki`·`ask_wiki`).

## 6. 사고 기록 — `docker compose config`가 시크릿을 뱉는다 (2026-07-14)

compose에 프로바이더 스위치를 배선한 뒤 **"실제로 전달되는지" 검증**하려고 `docker compose config`를 돌렸다. 검증 자체는 옳았고 결과도 맞았지만, 이 명령은 **`.env`를 치환해 출력**하므로 실제 API 키가 터미널에 그대로 찍혔다.

- **아이러니**: 같은 세션에서 `/brief` 커맨드에 *"키는 존재 여부만 확인하고 값은 절대 출력하지 말 것"*이라고 **직접 써놓고** 어겼다. 규칙을 아는 것과 그 규칙이 **어떤 명령에서 발동하는지 아는 것**은 다르다.
- **피해 범위**: 무료 티어 키(결제수단 미연결) · `.env`는 gitignore라 레포 미유입 · denylist에 등록된 **API 키 프리픽스 패턴**이 커밋 경로를 이미 막고 있었음(W-1). 노출은 세션 로그 한정 → **키 로테이션 권고**로 마무리.
- **교훈**: 시크릿 가드는 **커밋 경로**(denylist·훅)에 있었지만 **출력 경로**엔 없었다. W-1이 막은 건 "저장소로 새는 것"이지 "화면으로 새는 것"이 아니다. 검증 명령이 설정을 *렌더링*하는 종류라면(`config`·`render`·`describe`) 시크릿을 함께 뱉는다고 가정해야 한다.
- **규칙 추가**: 이 레포에서 compose 환경을 확인할 땐 `docker compose config | grep -vE 'KEY|TOKEN|PASSWORD'`로 거른다.
- **후일담 — 가드가 이 문서를 막았다**: 위 사고를 기록하며 denylist의 키 프리픽스를 **본문에 그대로 인용**했더니 pre-commit 훅이 커밋을 차단했다. `/record` 커맨드에 *"기록 문서에 금칙어 자체를 인용하지 말 것"*이라고 직접 써놓고 어긴 것이다. 자기 스캔은 *실제 키 패턴*(프리픽스+난수)을 찾았는데 훅은 *프리픽스 자체*를 본다 — **검사 대상이 서로 달랐다.** W-1의 가드는 이 한 세션에서만 두 번(컨벤션 금칙어·키 프리픽스) 실제로 일했다. **문서화된 규칙보다 실행되는 가드가 강하다**는 걸 만든 사람이 직접 증명한 셈.

## 7. 정직한 한계

- 사례 11건은 **한 레포·한 환경**(Windows + 이 샌드박스 구성)의 표본이다. "Codex가 UTF-8에 약하다"는 **이 환경에서의 실측**이지 일반 명제가 아니다.
- "로직은 안 틀렸다"는 **스펙이 정확했을 때**의 이야기다. 스펙이 모호했던 구간은 애초에 위임하지 않았으므로 이 표본엔 안 잡힌다 — 선택 편향이 있다.
- §3-6의 교훈은 **감사자도 AI**라는 한계를 안고 있다. 사람 검수를 대체하지 않으며, 실제로 라벨 판정은 연우님이 했다.
